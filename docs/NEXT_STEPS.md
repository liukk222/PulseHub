## 3. 架构与安全边界

- `pulsehub-agent.exe` 是配置文件和 HID 设备的唯一运行态所有者。
- `pulsehub-config.exe` 只通过 IPC 编辑/提交配置，不得直接打开 HID。
- GUI 托盘的“退出托盘”是用户退出整个 PulseHub 的入口：必须请求代理应用安全退出配置，然后让代理和 GUI 都退出。
- 安全退出配置由用户在 GUI 的“退出配置”页维护；默认仍为 DPI `1600`、`1000 Hz`，左键、右键、中键、G4、G5、G6 全部为鼠标原生功能。
- 退出前的设备修改仍由 `pulsehub-agent.exe` 执行，GUI 不得直接访问 HID；每项写入都必须回读校验并记录结果。
- 左键和右键必须继续受主点击保护。
- 普通自动化测试和 CI 禁止写入真实 HID 或板载闪存。
- 任何硬件写入工具必须保留显式确认参数，并在写入后回读校验。
- 不要移除板载写入的幂等、防抖、重连保护，也不要通过缩短延迟来做压力写入。
- G HUB 与 PulseHub 不做并发控制；执行 HID 实机测试前应退出 G HUB。
- 不记录设备序列号、用户路径中的敏感信息或原始完整 HID 数据。

## 4. 下一阶段目标：发布加固

下一阶段先修正“退出托盘”生命周期，再把当前可用程序变成可重复构建、可诊断、可安全安装的 Windows 应用。按以下顺序推进，每一项单独提交。

### 4.1 实现安全退出配置（已完成并通过 G102 实机验证）

1. 为 IPC 增加明确的代理关闭请求；只有当前用户会话内通过鉴权的配置程序可以调用。
2. 用户选择“退出托盘”且草稿处理完成后，GUI 请求代理进入 `ShuttingDown` 状态，不再接受配置提交或环境切换。
3. 代理把运行态 DPI 设置为 `1600`，并把左键、右键、中键、G4、G5、G6 全部恢复为原生鼠标按键。
4. 对 DPI 和按键配置执行回读校验；板载写入继续使用现有幂等、防抖和事务保护，不得通过重复写入强行恢复。
5. 恢复成功后刷新日志、关闭 HID/IPC/系统 hook，退出 `pulsehub-agent.exe`；GUI 确认代理退出后删除托盘并退出。
6. 设备未连接时无需等待设备出现，在有界超时后记录“未连接，无法恢复”并退出。设备在线但恢复失败时，GUI 显示失败原因并提供“重试恢复 / 仍然退出”，不能伪装成恢复成功。
7. 关闭主窗口和最小化窗口仍只隐藏/最小化 GUI，不应停止代理；只有显式“退出托盘”执行完整退出。

验收：退出前实测 DPI 为 `1600`，六个按键均为原生功能，随后 `pulsehub-agent.exe` 与 `pulsehub-config.exe` 都消失；重启 PulseHub 后仍可正常应用 Office/CS2 配置。

实现结果（2026-07-22）：新增 `shutdown` IPC、托盘退出进度界面、失败后的“重试恢复 / 仍然退出”、设备离线有界等待及强制审计日志。代理进入退出流程后会阻断前台切换事件，防止安全还原被 Office/CS2 再次覆盖。可重试 HID 超时最多自动尝试 3 次，总预算 10 秒；每次仍通过既有幂等和板载事务保护。实机只读回读确认 DPI `1600`，左/右/中/G4/G5 为原生鼠标按键，G6 为原生 DPI 切换，两个进程均正常退出。

### 4.2 建立 Release 基线

1. 运行完整质量门禁并记录结果。
2. 构建 `x86_64-pc-windows-msvc` Release 产物。
3. 确认 `pulsehub-agent.exe` 和 `pulsehub-config.exe` 使用 Windows GUI subsystem，启动时没有控制台窗口；`pulsehub-probe.exe` 保留控制台。
4. 记录可执行文件大小、依赖 DLL、启动结果和当前未解决的发布阻塞项。
5. 修正文档中仍与当前实现不符的历史描述，但不要借机重构业务逻辑。

验收：干净工作区可用一组固定命令生成可运行的 Release 二进制，Debug/Release 行为一致。

### 4.3 单实例与生命周期加固

1. 检查代理是否可靠保持每用户会话单实例。
2. 为托盘配置程序增加每用户会话单实例保护；第二次启动应激活现有窗口，不能生成第二个托盘或第二份草稿。
3. 验证 Explorer 重启和托盘图标重建；关闭主窗口时代理独立运行，显式退出托盘时代理安全退出。
4. 验证有未保存草稿时，“退出托盘”仍进入确认流程。

验收：反复启动程序只存在一个代理、一个配置进程和一个托盘图标，既不丢草稿也不重复控制设备；退出托盘后两个进程均结束。

### 4.4 性能与稳定性基线

1. 增加只读的 Windows 性能采样脚本，采集 agent 的 CPU、Private Working Set、线程数和句柄数。
2. Release 模式预热 5 分钟，再以 1 秒间隔采样 5 分钟，共 300 个样本。
3. 分别覆盖稳定 Office、稳定 CS2、GUI 隐藏、鼠标拔插恢复后的场景。
4. 结果写入 `docs/testing/`，不要把机器用户名、序列号或完整本地路径提交进仓库。

目标：平均 CPU `< 0.1%`，Private Working Set P95 `<= 15 MB`；如果未达标，先报告证据和热点，再决定优化方案。

### 4.5 安装与卸载方案

性能和单实例通过后，再选择安装器技术。首个安装方案必须支持：

- 每用户安装到 `%LOCALAPPDATA%\Programs\PulseHub`，无需管理员权限。
- 开始菜单入口和可选的 `HKCU Run` 登录启动。
- 升级前让进程正常退出，再替换二进制。
- 卸载时移除启动项和程序文件，并让用户选择是否保留配置与日志。
- 为后续代码签名保留明确步骤；没有证书时不得声称产物已签名。

## 5. 本轮建议立即执行的第一个任务

先完成“4.1 实现安全退出配置”，不要立刻编写安装器。修改前重点阅读代理退出顺序、IPC 和 G102 板载写入文档；实现完成后再运行：

~~~powershell
git status --short --branch
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo build --workspace --release --target x86_64-pc-windows-msvc
~~~

然后检查两个应用的入口属性、Release 进程行为及产物依赖，新增一份简短的 Release 基线报告。若构建失败，先定位真实原因；不要删除 `Cargo.lock`、跳过测试或修改 HID 协议来绕过构建问题。

## 6. 现实测试需要用户参与的项目

以下操作不能由普通自动化测试自行假定成功，完成实现后给用户明确命令和观察点：

- HID/DPI/按键或板载闪存写入。
- 鼠标拔插、Windows 注销/登录、睡眠恢复。
- Explorer 重启后的托盘恢复。
- CS2 前台切换与真实鼠标速度/按键行为。
- 安装、升级、卸载和登录启动。

用户已批准合理的 HID 读写测试，但每次仍要说明写入对象、是否涉及板载闪存以及恢复方式。
